Module Systems

01
CJS
CommonJS
Node.js original module system. Synchronous, dynamic. Default in .js files when package.json has no "type" or has "type": "commonjs".
const x = require('./foo') module.exports = { x }
ESM
ES Modules
The modern standard. Static, async, tree-shakeable. Native in browsers & modern Node. Use .mjs or set "type": "module" in package.json.
import { x } from './foo.js' export const y = 1
UMD
Universal Module Definition
Compatibility format for libraries. Works in browser globals, AMD, and CJS. Rarely written by hand — bundlers output it. Mostly legacy today.
// works as CJS, AMD, or // browser global window.Lib // output by bundlers only
The interop problem: CJS and ESM don't mix cleanly. CJS can't import ESM. ESM can only require() CJS via dynamic import workarounds. This is the source of most "ERR_REQUIRE_ESM" headaches. Libraries increasingly ship both formats (dual packages).
File extensions that matter: .mjs = always ESM · .cjs = always CJS · .js = depends on nearest package.json "type" field

Bundlers

02
source files → resolve imports → transform (via plugins) → tree-shake → bundle + minify → dist/
Webpack Industry Standard
The OG. Extremely powerful and configurable. Complex config. Dominates legacy and large enterprise codebases. CRA used it under the hood.
webpack.config.jsloaderspluginscode splitting
Vite Modern Default
Dev server uses native ESM — no bundling in dev, instant HMR. Rollup under the hood for production builds. The new React/Vue default.
vite.config.tsesbuild (dev)rollup (prod)HMR
Rollup Libraries
Best-in-class for library authoring. Excellent tree shaking, clean output. Outputs ESM/CJS/UMD. Vite uses it for production builds.
rollup.config.jsESM outputlibrary bundles
esbuild Speed
Written in Go. 10–100× faster than JS-based bundlers. No type checking. Used as the transformer inside Vite and many other tools.
no tsconfig emitstrip types onlyblazing fast
Parcel Zero Config
Zero config bundler. Auto-detects everything. Good for quick projects. Less common in production apps today.
zero configauto transforms
Turbopack Emerging
Rust-based, Vercel-built Webpack successor. Powers Next.js dev server. Still maturing. Watch this space.
Next.js 13+Rustincremental

TypeScript Build Tools

03
Key distinction: Some tools do type checking, some just strip types (transpile). They are different jobs. Speed vs safety tradeoff.
tsc Official
The TypeScript compiler. Type checks AND emits JS. Slow on large projects. Used for type checking in CI; often not used for bundling.
tsconfig.jsontype checkemit JSdeclaration files
ts-node Node Dev
Run TypeScript directly in Node without a build step. Compiles on the fly. Slow startup. Common for scripts and server dev.
on-the-fly compileno emitscripts
tsx ts-node replacement
Modern ts-node alternative using esbuild under the hood. Much faster startup. Type stripping only — no type checking.
esbuild backedfastno type check
tsup Library Build
Zero-config library bundler for TypeScript. Wraps esbuild. Outputs ESM + CJS + .d.ts files with minimal setup.
tsup.config.tsdual outputd.ts gen
tsbuildinfo Incremental
Not a tool — a file. Generated by tsc --incremental. Caches compilation state so tsc only recompiles changed files.
incremental buildscache file
SWC Rust
Rust-based TS/JS compiler. Used by Next.js to replace Babel. Strips types only — no type checking. Very fast.
.swcrcNext.js defaultno type check

Transpilers

04
Babel Legacy / React
Transforms modern JS/TS/JSX to older JS for browser compatibility. Huge plugin ecosystem. Slow. Being replaced by esbuild/SWC in most pipelines. Still needed for some custom transforms.
babel.config.js.babelrcpresetspluginsJSX
esbuild Both
Also acts as a transpiler (not just bundler). Transforms TS/JSX to JS extremely fast. Used directly or via Vite, tsx, tsup.
transform APIJSXtarget env
PostCSS CSS Only
CSS transpiler. Transforms CSS with plugins — autoprefixing, nesting, custom properties. Tailwind runs as a PostCSS plugin.
postcss.config.jsautoprefixerTailwind

Linting & Formatting

05
Linter vs Formatter: A linter catches code quality issues (unused vars, bad patterns). A formatter enforces style (indentation, quotes, semicolons). Don't use both for style — let Prettier own style, ESLint own logic.
ESLint Linter
The JS/TS linter. Catches bugs and bad patterns. Highly configurable via plugins. V9 introduced flat config (eslint.config.js), replacing the old .eslintrc.
eslint.config.js.eslintrctypescript-eslinteslint-plugin-react
Prettier Formatter
Opinionated code formatter. Reformats your code on save. Removes all style debates from teams. Configure it minimally and forget it.
.prettierrcprettier.config.js.prettierignore
Biome All-in-one
Rust-based. Does both linting and formatting in one tool. Very fast. Aims to replace ESLint + Prettier combo. Still gaining adoption.
biome.jsonRustESLint replacement
Stylelint CSS
Like ESLint but for CSS/SCSS. Catches CSS errors, enforces ordering, prevents bad patterns.
.stylelintrcCSS/SCSS
lint-staged + Husky Git Hooks
Husky wires up Git hooks. lint-staged runs linters only on staged files. Together they enforce quality on every commit.
.husky/pre-commitlint-staged config

Config Files Reference

06
tsconfig.json TypeScript compiler config. Controls what gets type-checked, what gets emitted, module format, path aliases, strictness. Core to any TS project. TypeScript
package.json More than dependencies. "type": "module" switches to ESM. "exports" field controls what your package exposes. "main" vs "module" for dual publish. Node / npm
vite.config.ts Vite's config. Define plugins, aliases (resolve.alias), proxy, build target, env vars. Very clean API. Vite
.eslintrc / eslint.config.js ESLint rules. Old format: .eslintrc.json/.yml/.js. New flat config (v9+): eslint.config.js. Defines rules, plugins, parser (use @typescript-eslint/parser for TS). ESLint
.prettierrc Prettier style rules: print width, tab width, single quotes, semicolons, trailing commas. Keep it minimal. Prettier
.editorconfig Editor-level defaults (indent size, line endings, charset). Respected by VS Code, JetBrains, etc. before any linter runs. Great for cross-IDE teams. Editor
postcss.config.js Configures PostCSS plugins. Typically just autoprefixer + Tailwind if you use it. PostCSS
.babelrc / babel.config.js Babel transform config. Defines presets (@babel/preset-env, @babel/preset-react, @babel/preset-typescript) and plugins. Babel
.nvmrc / .node-version Pins the Node.js version for the project. Used by nvm / Volta to auto-switch runtimes. Node
.env / .env.local Environment variables. .env is committed (no secrets). .env.local is gitignored. Vite uses VITE_ prefix. Node uses process.env. Runtime

JS Runtimes

07
V8 · 2009
Node.js
  • The default server runtime
  • CJS by default, ESM supported
  • Massive npm ecosystem
  • V8 engine
  • No built-in TS support
  • Node 22+ has --strip-types
V8 · 2020
Deno
  • Secure by default
  • Native TypeScript support
  • ESM-only, URL imports
  • Built-in formatter & linter
  • Smaller ecosystem
  • npm compatible since v1.28
JavaScriptCore · 2022
Bun
  • Extremely fast (JSC engine)
  • Built-in TS support
  • Built-in bundler & test runner
  • Drop-in Node replacement
  • Built-in package manager
  • Still maturing

Package Managers

08
npm Default
Comes with Node. Slowest of the three. package-lock.json for lockfile. Works everywhere — safest default for open source.
package-lock.jsonnode_modules
pnpm Monorepos
Faster, disk-efficient via content-addressable store. Strict dependency isolation. Excellent monorepo support. Growing default in enterprise and monorepo setups.
pnpm-lock.yamlworkspaceshoisting
yarn Legacy / PnP
Classic (v1) is still widespread. Yarn Berry (v3+) introduced Plug'n'Play (no node_modules). Largely replaced by pnpm in new projects.
yarn.lockPnPworkspaces
Monorepo tools on top: Turborepo, Nx, and Lerna orchestrate multi-package repos. They add task caching and parallelism on top of your package manager's workspaces. Your Audiophile project uses Turborepo — it handles build ordering, not bundling.
