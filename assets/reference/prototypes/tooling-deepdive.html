01

Build Pipeline & Bundlers

Priority

What a build pipeline actually does

Before a single bundler is involved, it helps to know what problem they're solving. Browsers can't run TypeScript. They can't resolve import x from './utils' without a file extension. They don't support path aliases like @/components. And shipping 400 separate files over HTTP/1.1 is slow.

A build pipeline transforms, resolves, and packages your source code into something browsers or Node can actually execute.

Source (.ts/.tsx) → Parse & Resolve → Transform → Tree-shake → Bundle → Minify → dist/

Not every tool does every step. Some only transform (Babel, SWC). Some only bundle (Rollup in basic mode). Modern tools like Vite split dev and prod paths entirely.

Webpack

The tool that defined the modern JS build. Everything is a module — JS, CSS, images, fonts. Webpack builds a dependency graph from your entry point and bundles everything according to that graph.

Core concepts you need to know:

Entry
Where the graph starts. Usually src/index.ts. Can have multiple entries for code splitting.
Output
Where bundles go. Filename templating: [name].[contenthash].js enables long-term caching.
Loaders
Transform files before they enter the graph. ts-loader or babel-loader for TS/JS. css-loader + style-loader for CSS. Runs right-to-left in the rules array.
Plugins
Operate on the whole bundle. HtmlWebpackPlugin injects script tags. MiniCssExtractPlugin extracts CSS to separate files.
Mode
"development" keeps source maps, skips minification. "production" enables tree-shaking, minification, optimizations.
Code splitting
Dynamic import() creates async chunks. SplitChunksPlugin auto-extracts shared vendor code.
Source maps
devtool: 'source-map' for prod, 'eval-source-map' for fast dev rebuilds.
// webpack.config.js — annotated skeleton
const path = require('path')
const HtmlWebpackPlugin = require('html-webpack-plugin')

module.exports = {
  mode: 'production',
  entry: './src/index.tsx',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: '[name].[contenthash].js',  // cache busting
    clean: true,
  },
  resolve: {
    extensions: ['.tsx', '.ts', '.js'],
    alias: { '@': path.resolve(__dirname, 'src') },
  },
  module: {
    rules: [
      {
        test: /\.tsx?$/,
        use: 'ts-loader',  // or babel-loader with @babel/preset-typescript
        exclude: /node_modules/,
      },
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader'],  // right-to-left
      },
    ],
  },
  plugins: [
    new HtmlWebpackPlugin({ template: './public/index.html' }),
  ],
  optimization: {
    splitChunks: { chunks: 'all' },  // extract vendor chunk
  },
}
⚠ Webpack Pain Points

Config is verbose and easy to get wrong. Loader order (right-to-left) trips everyone up. Cold start is slow on large projects. Webpack 5 improved things with persistent caching (cache: { type: 'filesystem' }) but it's still slower than Vite in dev.

Vite

Vite rethinks the dev/prod split entirely. The key insight: in development, you don't need to bundle at all. Modern browsers support native ES modules — so Vite serves your source files directly and lets the browser do the import resolution.

Dev mode: No bundle. Files are served as native ESM. Transforms happen on-demand per file (via esbuild — microseconds per file). HMR replaces only the changed module, not the whole bundle. Result: instant server start regardless of project size.

Production: Switches to Rollup for an optimized, tree-shaken, code-split bundle. Why not esbuild for prod? Rollup has more mature code splitting and plugin ecosystem.

// vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],

  resolve: {
    alias: { '@': '/src' },  // path aliases
  },

  server: {
    port: 3000,
    proxy: {
      '/api': 'http://localhost:4000',  // proxy to backend
    },
  },

  build: {
    outDir: 'dist',
    sourcemap: true,
    rollupOptions: {
      output: {
        manualChunks: {           // explicit vendor splitting
          vendor: ['react', 'react-dom'],
        },
      },
    },
  },

  // env vars: VITE_ prefix exposes to client
  // import.meta.env.VITE_API_URL
})
✓ Key Vite Concepts

Pre-bundling: On first start, Vite uses esbuild to pre-bundle node_modules into single ESM files. This is why it's fast even with many deps — they become single requests.

import.meta.env replaces process.env. Only vars prefixed with VITE_ are exposed to the browser. Server-only vars stay server-side.

Plugin compatibility: Vite plugins follow the Rollup plugin API. Most Rollup plugins work in Vite as-is.

Rollup

Rollup's strength is library output. It produces very clean, minimal bundles — no Webpack boilerplate runtime, just your code. It pioneered tree shaking (static analysis of ESM import/export to eliminate dead code).

// rollup.config.js — library build
import typescript from '@rollup/plugin-typescript'
import resolve from '@rollup/plugin-node-resolve'

export default {
  input: 'src/index.ts',
  output: [
    { file: 'dist/index.cjs', format: 'cjs' },  // for require()
    { file: 'dist/index.mjs', format: 'esm' },  // for import
    { file: 'dist/index.umd.js', format: 'umd', name: 'MyLib' },
  ],
  external: ['react'],  // don't bundle peer deps
  plugins: [resolve(), typescript()],
}
ℹ Dual Package Publishing

When building a library with both CJS and ESM output, your package.json needs the "exports" field to tell consumers which to use. Without it, older tools fall back to "main" (CJS) and newer tools use "module" (ESM).

// package.json — dual package setup
{
  "main": "./dist/index.cjs",    // CJS fallback
  "module": "./dist/index.mjs",  // ESM for bundlers
  "exports": {
    ".": {
      "import": "./dist/index.mjs",   // ESM consumers
      "require": "./dist/index.cjs"  // CJS consumers
    }
  },
  "types": "./dist/index.d.ts"    // TypeScript types
}

esbuild

esbuild is written in Go and parallelizes work across CPU cores. It's 10–100× faster than JS-based tools. It does not type-check — it strips TypeScript syntax and moves on. This is intentional. Type checking is a separate concern.

You rarely configure esbuild directly — it's the engine inside Vite (dev transforms), tsx, tsup, and other tools. But knowing its model helps you understand why those tools are fast.

// esbuild direct usage (rare, but useful for scripts)
import * as esbuild from 'esbuild'

await esbuild.build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  platform: 'node',          // or 'browser'
  target: 'node18',          // transpile target
  format: 'esm',
  outfile: 'dist/index.js',
  sourcemap: true,
})
⚠ What esbuild Cannot Do

No TypeScript type checking. No decorator metadata emission (required by some NestJS features — needs Babel or tsc for this). Limited code splitting compared to Rollup. Not a full Webpack replacement for complex apps.

TypeScript Build Tools

Priority
⚡ The Core Distinction

Type checking = verifying your types are correct. Slow. Only tsc does this.

Transpiling = stripping TypeScript syntax and outputting JS. Fast. esbuild, SWC, Babel, tsx all do this — but they don't check your types. They'll happily compile broken TypeScript.

Production pipelines often do both: fast transpiler for the build, tsc --noEmit in CI for type safety.

tsc — The TypeScript Compiler

The official compiler from Microsoft. It does two things: type-checks your code and emits JavaScript output. On large codebases it's slow because it does both together.

Key CLI flags:

--noEmit
Type-check only, don't output files. Used in CI alongside a faster bundler.
--watch
Recompile on file changes. Useful for libraries or Node servers without a bundler.
--incremental
Saves compile state to .tsbuildinfo. Only recompiles changed files. Major speedup on large projects.
--build (-b)
Project references mode. Builds multiple tsconfig files in dependency order. Used in monorepos.
--project (-p)
Points to a specific tsconfig file, e.g. tsc -p tsconfig.build.json.

ts-node

Runs TypeScript files directly in Node.js without a build step. Registers a require hook that compiles TypeScript on-the-fly using tsc. Startup is slow (compiles on demand). Good for scripts, migrations, one-off tools.

# Run a TS file directly
npx ts-node src/seed.ts

# Skip type checking for speed
npx ts-node --transpile-only src/seed.ts

# ESM mode (newer projects)
npx ts-node --esm src/server.ts
ℹ ts-node config in tsconfig

You can configure ts-node inline in tsconfig.json under a "ts-node" key — useful for setting transpileOnly: true globally for your dev server.

tsx

Modern replacement for ts-node. Uses esbuild under the hood — startup is near-instant. Type stripping only, no type checking. Supports both CJS and ESM. The recommended default for running TS scripts today.

# Drop-in ts-node replacement
npx tsx src/server.ts
npx tsx watch src/server.ts  # watch mode

# As a Node loader (Node 18.19+)
node --import tsx/esm src/server.ts

tsup

The go-to for building TypeScript libraries. Wraps esbuild with sensible defaults. Outputs CJS, ESM, and declaration files (.d.ts) in one command. Minimal config needed.

// tsup.config.ts
import { defineConfig } from 'tsup'

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['cjs', 'esm'],   // dual output
  dts: true,               // generate .d.ts files
  sourcemap: true,
  clean: true,             // clear dist/ before build
  splitting: false,        // set true for code splitting in ESM
  treeshake: true,
})
✓ tsup vs tsc for libraries

Use tsup for the build output (fast, dual format). Run tsc --noEmit separately for type safety. Many library setups do both in CI: "build": "tsup && tsc --noEmit".

SWC

Rust-based compiler. Replaces Babel in Next.js by default since Next 12. Strips types and transforms syntax extremely fast. Config via .swcrc. Like esbuild — no type checking, just transform.

// .swcrc
{
  "jsc": {
    "parser": {
      "syntax": "typescript",
      "tsx": true,
      "decorators": true  // needed for NestJS
    },
    "target": "es2020",
    "transform": {
      "legacyDecorator": true,
      "decoratorMetadata": true  // required for NestJS DI
    }
  },
  "module": { "type": "commonjs" }
}
⚠ NestJS + SWC

NestJS relies on decorator metadata for its dependency injection system. esbuild can't emit this metadata. SWC can (decoratorMetadata: true). This is why NestJS docs recommend SWC over esbuild for fast development builds.

03

Config Files

Priority

tsconfig.json — Deep Dive

The most important config file in any TypeScript project. Controls the compiler, shapes what errors you get, determines output format. Many developers only set strict: true and move on — there's a lot more here.

Compiler Options — Core

target
JS version to emit. "ES2020", "ES2022", "ESNext". Affects which syntax is downleveled. Does not affect what features are available — that's lib.
module
Module format for emitted JS. "CommonJS" for Node CJS. "ESNext" or "ES2022" for ESM. "NodeNext" for Node with proper ESM interop (requires extensions in imports).
moduleResolution
How imports are resolved. "node" (classic). "bundler" (Vite/esbuild — allows extensionless imports). "NodeNext" (strict ESM, requires .js extensions in imports).
lib
Built-in type declarations to include. ["DOM", "ES2022"] for browser apps. ["ES2022"] for Node-only. Without DOM, you won't have types for window, fetch, etc.
outDir
Where compiled JS goes. Often "dist". Ignored by Vite (it uses Rollup for output) but used by tsc directly.
rootDir
Root of source files. tsc mirrors the directory structure relative to this in outDir.
declaration
true generates .d.ts files. Required when publishing a library so consumers get types.
declarationMap
Generates .d.ts.map files. Lets IDE "Go to Definition" jump to original TS source instead of compiled .d.ts. Very useful in monorepos.
sourceMap
Generate source maps. Lets debuggers map JS back to TS. Always enable for server-side code.

Compiler Options — Strictness

strict
Master switch enabling: noImplicitAny, strictNullChecks, strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, noImplicitThis. Always use this.
noUncheckedIndexedAccess
Array index access returns T | undefined instead of just T. More accurate but more verbose. Not included in strict.
exactOptionalPropertyTypes
Prevents assigning undefined to optional properties explicitly. Stricter than default.
noImplicitReturns
Error if a function doesn't return in all code paths.
noUnusedLocals / noUnusedParameters
Errors on unused variables/parameters. Keeps code clean. Can be annoying during active development — some teams leave these off locally.

Compiler Options — Interop & Paths

esModuleInterop
Enables import React from 'react' style (default imports from CJS modules). Nearly always true. Works with allowSyntheticDefaultImports.
paths
Path aliases for TypeScript: { "@/*": ["./src/*"] }. Important: TypeScript resolves these for type checking only. Your bundler (Vite, Webpack) also needs to know about them separately.
baseUrl
Required when using paths. Sets the base for relative path resolution.
resolveJsonModule
Allows importing .json files with types inferred from their shape.
skipLibCheck
Skip type checking of .d.ts files in node_modules. Almost always true — you don't want to fail your build because of a badly typed third-party lib.

includes / excludes / references

include
Glob patterns for files to compile. Default is all .ts/.tsx in the project. Usually ["src/**/*"].
exclude
Files to skip. node_modules is excluded by default. Add dist, **/*.spec.ts if needed.
extends
Compose configs. Common pattern: a base tsconfig.base.json with shared settings, extended by tsconfig.json (app) and tsconfig.build.json (emit-only, excludes tests).
references
Project references for monorepos. Let tsc --build compile packages in dependency order with incremental caching.

Real-World tsconfig Patterns

// tsconfig.json — Vite + React app
{
  "compilerOptions": {
    "target": "ES2020",
    "lib": ["ES2020", "DOM", "DOM.Iterable"],
    "module": "ESNext",
    "moduleResolution": "bundler",  // Vite-specific
    "jsx": "react-jsx",             // No React import needed
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noFallthroughCasesInSwitch": true,
    "allowImportingTsExtensions": true,
    "resolveJsonModule": true,
    "isolatedModules": true,  // required for esbuild/Vite
    "noEmit": true,           // Vite handles emit, not tsc
    "esModuleInterop": true,
    "skipLibCheck": true,
    "paths": { "@/*": ["./src/*"] }
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}
// tsconfig.json — NestJS (Node.js backend)
{
  "compilerOptions": {
    "module": "CommonJS",
    "target": "ES2021",
    "lib": ["ES2021"],
    "moduleResolution": "node",
    "outDir": "./dist",
    "sourceMap": true,
    "declaration": true,
    "strict": true,
    "emitDecoratorMetadata": true,  // REQUIRED for NestJS DI
    "experimentalDecorators": true, // REQUIRED for NestJS
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}
⚠ isolatedModules: true

Required when using esbuild or SWC (including Vite). These tools compile each file independently — they can't see the full type graph. isolatedModules: true makes tsc warn you when you use patterns that require cross-file analysis (e.g. const enum, re-exporting types without the type keyword).

ESLint Config

ESLint v9 changed the config system significantly. You'll encounter both formats in the wild.

Old Format — .eslintrc.* (v8 and below)

// .eslintrc.json
{
  "parser": "@typescript-eslint/parser",
  "parserOptions": {
    "ecmaVersion": "latest",
    "sourceType": "module",
    "project": "./tsconfig.json"  // enables type-aware rules
  },
  "plugins": ["@typescript-eslint", "react-hooks"],
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended",
    "plugin:react-hooks/recommended"
  ],
  "rules": {
    "@typescript-eslint/no-explicit-any": "warn",
    "@typescript-eslint/consistent-type-imports": "error"
  },
  "ignorePatterns": ["dist", "node_modules"]
}

New Format — eslint.config.js (v9 flat config)

The new format is a JS array of config objects. No more cascading directory configs. More explicit, more predictable.

// eslint.config.js (ESLint v9)
import js from '@eslint/js'
import tseslint from 'typescript-eslint'
import reactHooks from 'eslint-plugin-react-hooks'
import globals from 'globals'

export default tseslint.config(
  js.configs.recommended,
  ...tseslint.configs.recommended,

  {
    languageOptions: {
      globals: { ...globals.browser, ...globals.es2022 },
      parserOptions: {
        project: true,  // use nearest tsconfig.json
        tsconfigRootDir: import.meta.dirname,
      },
    },
    plugins: { 'react-hooks': reactHooks },
    rules: {
      ...reactHooks.configs.recommended.rules,
      '@typescript-eslint/no-explicit-any': 'warn',
      '@typescript-eslint/consistent-type-imports': 'error',
      '@typescript-eslint/no-unused-vars': ['error', {
        argsIgnorePattern: '^_'  // allow _unused params
      }],
    },
  },

  {
    ignores: ['dist/**', 'node_modules/**'],
  }
)
ℹ Type-Aware Rules

When you pass project: true to parserOptions, ESLint loads your TypeScript type information. This enables more powerful rules like @typescript-eslint/no-floating-promises and await-thenable that need to understand types. These are slower but catch real bugs.

Useful ESLint Rules for TS Projects

consistent-type-imports
Enforce import type { Foo } for type-only imports. Helps bundlers/esbuild eliminate them cleanly.
no-floating-promises
Disallow unawaited async calls. One of the most useful type-aware rules — prevents silent async bugs. Needs project set.
no-misused-promises
Prevents passing async functions where void-returning ones are expected (e.g. event handlers that should return void).
prefer-nullish-coalescing
Prefer ?? over || when dealing with null/undefined.

Prettier Config

Prettier is deliberately opinionated. The point is to end all style debates — configure it once, commit the config, and forget it.

// .prettierrc
{
  "semi": false,              // no semicolons
  "singleQuote": true,       // 'strings' not "strings"
  "trailingComma": "all",    // trailing commas everywhere valid
  "printWidth": 100,         // line length target (not a hard limit)
  "tabWidth": 2,
  "arrowParens": "always",  // (x) => x, not x => x
  "endOfLine": "lf"          // consistent line endings cross-platform
}
⚠ ESLint + Prettier Conflict

ESLint has style rules that conflict with Prettier. The fix: install eslint-config-prettier and add "prettier" as the last item in your extends array (or spread it last in flat config). This disables all ESLint rules that Prettier handles. Never use eslint-plugin-prettier (runs Prettier as a linting rule — slow and gives ugly error output).

// .prettierignore
dist
node_modules
*.min.js
public/
# Don't format generated files

package.json — Beyond Dependencies

Most developers only use dependencies, devDependencies, and scripts. The fields that actually matter for module resolution and publishing:

{
  "name": "my-lib",
  "version": "1.0.0",

  // Tells Node.js how to treat .js files in this package
  "type": "module",  // "module" = ESM default | "commonjs" = CJS default

  // Modern entrypoints (takes precedence over "main"/"module")
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    },
    // Named exports for subpaths
    "./utils": "./dist/utils.js"
  },

  // Legacy fallbacks (older bundlers that don't read "exports")
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "types": "./dist/index.d.ts",

  // What to include when you npm publish
  "files": ["dist", "README.md"],

  // Tooling that lives in package.json
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"]
  }
}

Often overlooked. Sets editor defaults that apply before any linter or formatter runs. Ensures consistent indentation and line endings even if a team member hasn't installed Prettier.

.env Files

Environment variable files. Loaded by dotenv (Node), Vite, or Next.js. Key rule: never commit secrets.

# .env            — committed, no secrets. Defaults.
# .env.local       — gitignored. Local overrides.
# .env.development — loaded in dev mode only
# .env.production  — loaded in prod mode only
# Priority (Vite): .env.local > .env.development > .env

# Vite — only VITE_ prefixed vars reach the browser
VITE_API_URL=http://localhost:4000
VITE_APP_NAME=MyApp

# Server-only (not exposed to browser in Vite)
DATABASE_URL=postgresql://...
JWT_SECRET=secret123
✓ Type-safe env vars

Use vite-plugin-env-compatible or define augmentations for import.meta.env in a vite-env.d.ts file to get autocomplete and type errors on unknown env vars.

Module Systems

Core

CommonJS (CJS)

Node's original module system, designed for synchronous server-side file loading. require() is synchronous and dynamic — you can call it anywhere, with computed strings.

// CJS — dynamic, synchronous
const path = require('path')
const { readFile } = require('fs/promises')

// Can be dynamic
const plugin = require(`./plugins/${name}`)

module.exports = { readFile }
module.exports.default = readFile  // named + default

Active when: file has .cjs extension, or .js and nearest package.json has no "type" or has "type": "commonjs".

ES Modules (ESM)

The TC39 standard. Static imports — the module graph is resolved before any code runs. This is what enables tree shaking. Native in browsers. Supported in Node 12+.

// ESM — static, async-capable
import { readFile } from 'fs/promises'
import type { User } from './types.js'  // type-only, erased at runtime

// Dynamic import (lazy-loaded, returns Promise)
const { heavy } = await import('./heavy-module.js')

export { readFile }
export default readFile
⚠ ESM in Node.js — The Extension Rule

When using "moduleResolution": "NodeNext" in tsconfig, imports must include the .js extension — even when importing a .ts file. You write import { x } from './foo.js', tsc resolves it to foo.ts at build time, and emits ./foo.js. This trips everyone up at first.

CJS ↔ ESM Interop

This is where real pain lives. The rules:

ESM → CJS
ESM can import CJS. The CJS module's module.exports becomes the ESM default export. Named exports are sometimes inferred from static analysis.
CJS → ESM
CJS cannot require() an ESM module. This is the source of ERR_REQUIRE_ESM. You can use dynamic import() as a workaround in async contexts.
Dual packages
Libraries shipping both formats use "exports" in package.json with "import" and "require" conditions. The consumer's bundler picks the right one.

UMD

A wrapper format that detects the environment and exposes itself as a CJS module, AMD module, or browser global. Generated by bundlers for legacy compatibility. You almost never write it by hand — it's a build output format. Mostly legacy.

The key: don't use ESLint for style. Let Prettier own formatting. Use eslint-config-prettier to disable any ESLint rules that overlap with Prettier. This prevents conflicts and keeps linting fast.

Git Hooks — Husky + lint-staged

Enforces quality at commit time. Husky hooks into Git. lint-staged filters the hook to only run on staged (changed) files — so you're not linting your entire codebase on every commit.

# Setup
npx husky init
npm install --save-dev lint-staged
// package.json
{
  "lint-staged": {
    "*.{ts,tsx,js}": [
      "eslint --fix --max-warnings 0",
      "prettier --write"
    ],
    "*.{json,md,yml}": ["prettier --write"]
  }
}
# .husky/pre-commit
npx lint-staged
✓ Also useful: commit-msg hook

Add Conventional Commits validation via commitlint. Enforces feat:, fix:, chore: prefix format. Required if you use automated changelog generation or semantic versioning.

Biome — The All-in-One Alternative

Biome is a Rust-based tool that replaces both ESLint and Prettier. Single binary, single config, much faster. It's not a complete ESLint replacement yet (fewer rules, fewer plugins), but for new projects it's worth considering.

// biome.json
{
  "$schema": "https://biomejs.dev/schemas/1.8.3/schema.json",
  "formatter": {
    "enabled": true,
    "indentWidth": 2,
    "lineWidth": 100
  },
  "linter": {
    "enabled": true,
    "rules": { "recommended": true }
  },
  "javascript": {
    "formatter": { "quoteStyle": "single" }
  }
}

Bun is worth watching for your NestJS work. It runs Node-compatible code, understands TypeScript natively, and has a built-in package manager that's significantly faster than npm. Bun as a runtime for NestJS is increasingly practical in 2024+.

ℹ Node.js --strip-types (v22.6+)

Node 22.6 added experimental --strip-types flag — run TypeScript directly without a compiler, similar to tsx/ts-node but built-in. It's just type stripping (no checking), and some syntax like decorators still need Babel/SWC. But it signals the direction: TS as a first-class Node.js citizen.

pnpm — Why It Matters

pnpm stores packages once in a global content-addressable store, then symlinks them into each project's node_modules. If 10 projects use React 18, it's stored once. This makes it significantly faster to install and uses far less disk.

It also enforces strict dependency isolation: you can only import packages that are in your own package.json — not packages that a dependency of yours happened to install. This catches a class of bugs where you accidentally depend on a transitive dependency.

# pnpm workspace setup (pnpm-workspace.yaml)
packages:
  - 'apps/*'
  - 'packages/*'

# Install in specific workspace
pnpm --filter my-app add react

# Run script in all packages
pnpm --recursive run build

Monorepo Tools

Ecosystem
ℹ Monorepo vs Package Manager Workspaces

Workspaces (npm/pnpm/yarn) handle installing packages together. Monorepo tools (Turborepo, Nx) handle building, testing, and running tasks efficiently across packages. They're different layers. Your Audiophile project uses Turborepo on top of npm/pnpm workspaces.

Turborepo

Adds a task orchestration and caching layer. Knows the dependency graph between packages, runs tasks in parallel where possible, and caches outputs. If nothing changed in packages/ui, its build is a cache hit — instant.

// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],  // build deps first (^)
      "outputs": ["dist/**"]   // what to cache
    },
    "dev": {
      "cache": false,          // don't cache dev server
      "persistent": true
    },
    "lint": {
      "dependsOn": ["^build"]
    },
    "test": {
      "dependsOn": ["build"],
      "outputs": ["coverage/**"]
    }
  }
}

Nx

More opinionated than Turborepo. Has generators (scaffold new apps/libs), a plugin system, and a dependency graph visualizer. Popular in enterprise Angular/React monorepos. XM Cyber's stack uses Nx — it goes deeper on affected-only builds (only build/test what's affected by your changes).
